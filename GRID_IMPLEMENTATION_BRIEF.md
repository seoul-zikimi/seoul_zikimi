## 2. 핵심 아키텍처 분기 — 그리드는 "두 개"다 ⚠️

기획서의 그리드 시스템은 **성격이 전혀 다른 두 곳**에서 등장합니다. 이걸 하나로 합치려 하면 막힙니다.

| | (A) 정답 맵 제작 그리드 | (B) 런타임 플레이 그리드 |
|---|---|---|
| 언제 | 개발 중(에디터), 디자이너가 정답 건물 배치 | 게임 중, 플레이어가 재료 배치 |
| 도구 | **Autotiles3D** (← 여기서만 사용) | **자체 런타임 그리드 시스템** (NGO 동기화) |
| 결과물 | `MapAnswerData` (정답 격자 데이터) | `RuntimeGrid` (현재 건축 상태) |
| 좌표계 | `Vector3Int` 셀 좌표 | **(A)와 동일한 좌표 규약** ← 채점 1:1 비교를 위해 필수 |

> **왜 Autotiles3D를 런타임에 못 쓰나:** 매뉴얼 4장 명시 — *"Autotiles 3D is designed as an editor-only, non-runtime tool."* Undo API 의존, 프리팹 재인스턴스화 로직 등 에디터 전용. 따라서 **Autotiles3D = 맵 오서링(정답 데이터 생성) 전용**, 런타임 배치는 자체 구현.

**플랜이 확보해야 할 핵심:** (A)와 (B)가 **같은 셀 좌표·같은 재료 ID·같은 회전 규약**을 공유하도록 데이터 모델을 먼저 통일할 것.

## 3. 데이터 모델 (먼저 확정해야 할 것)

### 3.1. 그리드 셀
3D 격자, 기본 단위 = 정사각형 큐브. 각 셀이 보유해야 하는 정보(기획서 맵 1.1):
1. 오브젝트 배치 여부
2. 배치된 오브젝트 종류(재료 ID)
3. 배치 방향(회전)
4. 완료된 공정 목록 — **2개 이상 누적 가능** (예: 배치 → 고정 → 페인트칠)

```
CellState {
  bool occupied;
  MaterialId materialId;
  Quaternion / int rotationStep;   // 90도 스텝 권장
  List<ProcessType> completedProcesses;  // 고정, 페인트칠 ...
  ulong ownerObjectId;  // 멀티칸 오브젝트의 앵커 참조용
}
```

### 3.2. 재료(오브젝트) 정의 — `MaterialDef` (ScriptableObject)
각 오브젝트는 **두 종류의 형상 정보**를 가짐(기획서 맵 1.2):
- **그리드 점유 데이터**: 칸 단위(예: 1x1x2, 1x3x2). 배치·채점 로직이 사용. **메시 모양과 무관**. 곡선(아치)도 정해진 칸 단위로 점유.
- **물리 콜라이더**: 보이는 메시 형태에 맞춤. 플레이어 이동·충돌 판정이 사용.

```
MaterialDef {
  MaterialId id;
  Vector3Int footprint;          // 점유 칸 (예: (1,3,2))
  GameObject prefab;             // 비주얼 + 콜라이더
  List<ProcessType> requiredProcesses;  // 이 재료에 요구되는 공정 (예: 기둥=고정, 벽=고정+페인트)
  bool mustBeFixed;              // 하중 부재 여부 (기둥/벽=true, 지붕=false)
  bool isBreakable;              // 유리 등
  int maxSpawnCount;             // 스폰 제한 (유리류는 무제한)
}
```

> 공정 요구는 **재료 타입 단위**로 정의 → 정답 데이터는 셀당 "재료 + 회전"만 저장하면 되고, 요구 공정은 `MaterialDef`에서 파생. 데이터 중복 제거.

### 3.3. 정답 맵 데이터 — `MapAnswerData` (ScriptableObject / JSON)
```
MapAnswerData {
  Vector3Int gridSize;
  Dictionary<Vector3Int, AnswerCell> cells;   // 정답: 위치별 재료ID + 회전
  Vector3 startPilePosition;                   // 시작 재료 더미 위치
  float timeLimitSeconds;
  Sprite answerImage;                          // 상시 확인용 완성 이미지(정답)
}
AnswerCell { MaterialId materialId; int rotationStep; }
```

## 4. 요구사항 — 시스템별 상세

### A. 맵 제작 / 정답 데이터 파이프라인 (Autotiles3D 활용 핵심)
기획서 맵 3장 요구사항:
- 유니티 씬에서 재료 프리팹을 3D 그리드에 직접 배치한다.
- 그 배치 결과가 해당 맵의 **정답 격자 데이터로 저장**된다.
- 저장된 정답 데이터에서 "각 칸의 정답 재료·정답 상태"가 자동 추출되어야 한다.
- 맵마다 전체 그리드 크기, 시작 재료 더미 위치, 제한시간 등을 설정 가능.

→ **이 전체가 Autotiles3D + 자체 익스포터로 구현됨. 자세한 설계는 7장.**

### B. 런타임 빌드 시스템 (플레이 중 배치)
- 플레이어가 든 재료를 그리드 셀에 배치 → `RuntimeGrid` 갱신 + 프리팹 스폰(네트워크).
- 멀티칸 오브젝트(1x3x2 등)는 앵커 셀 + footprint로 점유 칸 전체를 차지. 겹침 검사.
- 배치 가능 여부: 셀 비어있음 + footprint 전체가 그리드 범위 내 + 충돌 없음.

### C. 건축 공정(상호작용) 단계 (건축·물리 1장)
재료별 공정 단계 예시(3단):
1. **배치됨** — 그리드에 놓임. 아직 불안정.
2. **고정됨** — 고정 상호작용 완료. 무너지지 않음. ('취소' 상호작용으로 고정 해제 가능)
3. **마감됨** — 페인트칠 등 마감 완료. 완성.

규칙:
- **하중을 견디는 오브젝트(기둥, 벽)만 고정 필수.** 지붕 등은 쌓아 올림(고정 불필요).
- 고정 안 해도 되는 걸 고정해도 **감점 없음**.
- 어떤 오브젝트에 어떤 공정이 필요한지는 **TAB 키**로 확인(오버쿡드 레퍼런스).
- **공정 적용 방식은 미결정(9장 참고)** — 기본 채택안: "필요한 도구를 가져와서 작업"(망치=고정, 페인트통=페인트). 협동 재미 ↑. 단점: 도구 종류·관리 증가.

### D. 플레이어 조작 (조작·카메라 1장)
- **W/A/S/D**: 이동(카메라 기준 상하좌우).
- **Space**: 물체 들기/내려놓기(토글).
- **E (약 3초 길게)**: 상호작용. **자동 선택** — 미고정 → 고정, 고정됐지만 미도색 → 페인트.
- **Shift**: 달리기.
- **Tab**: 각 오브젝트에 필요한 공정 확인.
- (레퍼런스 정신상) 재료 **던지기** 상호작용도 협동 재미의 핵심 → 들기 시스템에 던지기 포함 고려.

### E. 카메라 (조작·카메라 2장)
- 플레이어를 **약간의 딜레이를 두고 부드럽게 따라감**(고정 추적 X). → **Cinemachine의 댐핑(damping) 추적**으로 구현(기획서가 명칭을 못 찾던 그 효과 = smooth/damped follow).
- 기본 각도: **쿼터뷰** (플레이어 기준 위·오른쪽으로 약 30도 내려다봄).
- **우클릭 드래그**: 카메라 회전.
- **마우스 스크롤**: 확대/축소.
- 멀티: 플레이어별 카메라(분할 아님, 각자 자기 화면). 대결 모드만 분할(후순위).

### F. 무너짐 — 규칙 기반 연쇄 (건축·물리 2장)
**물리 엔진 미사용. 상태값만 동기화.**
- **트리거**: 미고정(배치됨) 재료가 외부 충격(플레이어 부딪힘 / 다른 재료에 맞음)을 받거나, 미고정 오브젝트 위에 다른 오브젝트가 놓이면 무너짐.
- **연쇄**: 한 재료가 무너지면 그 위에서 지지받던 재료들도 지지를 잃음 → "고정됨"이 아니면 함께 무너짐.
- **처리**: 그리드에서 제거 → 바닥에 떨어짐 → 다시 주워서 재배치 가능.
- → 결정론적 지지 그래프(support graph) 계산 + 서버 권위 + 상태 브로드캐스트로 설계.

### G. 채점 시스템 (맵 2장)
게임 종료 시 `RuntimeGrid` vs `MapAnswerData` 비교. **정답=100% 기준, 미충족분만큼 감점하는 방식**(개요서/게임플레이 명시).
| 항목 | 검사 | 점수(변경 가능) |
|---|---|---|
| 배치 정확도 | 셀별 재료·방향이 정답과 일치 | 올바른 칸당 +200 |
| 공정 완료 | 셀별 요구 공정 정상 완료 | 올바른 칸당 +100 |
| 조기 종료 보너스(옵션) | 제한시간보다 빨리 종료 | 일찍 끝낸 초당 +10 |

### H. 게임 루프 & 종료 (게임플레이 1장)
**플레이 사이클:**
`(멀티: 팀 메이킹) → 건축물 선택 → 건축 + 타이머 시작 → 시간 종료 또는 '건축 종료' → 채점 → 보상`

**메인 루프:**
`타이머 시작 + 정답 이미지 확인(상시) → 재료 운반 → 배치 → 공정(고정/페인트) → 완성`

**종료 버튼 2종 분리:**
- **게임 나가기**(설정 UI): 채점 없이 떠남.
- **건축 종료**: 타이머 종료 전 조기 종료 + 채점 요청.
  - 멀티: 개인 이탈 가능. 마지막 1인 남으면 혼자 종료 가능. **혼자 남아 '나가기' 하면 게임 무효(채점·보상 없음).**
  - 멀티 '건축 종료'는 **팀원 전원 동의** 필요 → 토글 동의 + "3/4 동의" UI 표시.

### I. 게임 모드 (게임플레이 2장)

## 7. Autotiles3D 활용 설계 (가장 중요) 🔧

**Autotiles3D는 4장 'A. 맵 제작 파이프라인'에서만 사용한다.** 런타임 코드에 포함하지 않는다.

### 7.1. 기획 요구 ↔ Autotiles3D 기능 매핑
| 기획 요구사항 | Autotiles3D 대응 | 비고 |
|---|---|---|
| 3D 그리드에 프리팹 배치 | `Autotiles3D_Grid` + `Autotiles3D_TileLayer` | Unit Scale = 큐브 1칸 크기 |
| 재료 프리팹 카탈로그 | `Autotiles3D_TileGroup` (재료별 Tile 등록) | **런타임 `MaterialDef`와 같은 프리팹 공유** |
| 씬에서 직접 배치/수정/삭제 | 에디터 컨트롤(좌클릭 배치, 우클릭 제거, Ctrl+휠 회전, Shift+드래그 돌출) | 디자이너 워크플로우 |
| 방향(회전) 정보 저장 | 타일 회전(Y축 90도 스텝) | `rotationStep`으로 매핑 |
| 배치 결과 → 정답 격자 데이터 | `Autotiles3D_TileLayer.InternalNodes` (`Dictionary<Vector3Int, InternalNode>`) **읽기** | 매뉴얼: 읽기는 안전, **쓰기는 비권장** |
| 좌표 변환 | `Autotiles3D_Grid.ToWorldPoint / ToWorldDirection`, `GetBlocks(Vector3Int)`, `IsExceedingLevelGrid` | 공개 API |

### 7.2. 정답 데이터 익스포터 (자체 에디터 스크립트)
1. 디자이너가 Autotiles3D로 **정답 건물**을 그리드에 배치(각 타일 = 재료 1종).
2. 에디터 메뉴 버튼(예: `Tools/MapExporter/Export Answer`) 실행.
3. `TileLayer.InternalNodes`를 순회 → 각 `Vector3Int` 셀에서 어떤 타일(=재료 ID)·회전인지 읽음.
4. 멀티칸 오브젝트 처리: 타일은 앵커 1칸으로 두고, `MaterialDef.footprint`로 점유 칸을 익스포터가 확장 계산(아치 등 곡선 비주얼도 칸 단위로 점유).
5. `MapAnswerData`(ScriptableObject 또는 JSON)로 직렬화: `gridSize`, 셀별 `{materialId, rotationStep}`, `startPilePosition`, `timeLimitSeconds`, `answerImage`.
6. 요구 공정은 저장하지 않음 → 런타임에서 `MaterialDef.requiredProcesses`로 파생.

### 7.3. 재료 ID 매핑 규약
- Autotiles3D Tile ↔ `MaterialDef.id` 를 **1:1로 묶는 매핑 테이블**을 만들 것(예: Tile 이름 = MaterialId, 또는 별도 lookup ScriptableObject).
- 이게 (A) 정답과 (B) 런타임이 같은 재료를 가리키게 하는 연결고리.

### 7.4. 주의/한계 (매뉴얼 5장)
- Autotiles3D는 Undo 등록이 비싸 대량 프리팹 인스턴스화 시 느려질 수 있음 → 프로젝트 설정에서 Undo 끄기(기본·권장).
- `InternalNodes`에 **직접 쓰지 말 것**. 읽기 전용으로만 사용.
- 베이킹(mesh combine)은 정답 맵 비주얼 최적화엔 쓸 수 있으나, 익스포터는 베이킹 전 원본 노드 데이터를 읽어야 함.

## 8. 권장 Phase 구조 (클로드 코드가 다듬을 초안)
- **Phase 0** — 데이터 모델 확정(3장), 재료 ID 매핑 규약, 좌표 규약 통일.
- **Phase 1** — Autotiles3D 세팅 + 정답 데이터 익스포터(7장). 샘플 맵 1개 정답 데이터 생성·검증.
- **Phase 2** — 싱글(로컬) 런타임 그리드 + 배치 시스템(B) + 플레이어 조작/카메라(D, E).
- **Phase 3** — 공정 시스템(C) + 도구 들기/던지기 + TAB UI.
- **Phase 4** — 채점(G) + 게임 루프/종료(H) + 타임어택 모드(I-1). **여기까지 = MVP.**
- **Phase 5** — NGO 멀티 동기화(5장) + 무너짐 규칙 연쇄(F) + 동의 종료 UI.
- **Phase 6+** — 재료 스폰 확정 구현(J), 대결 모드, 자유 모드.

## 9. 미해결 결정사항 (기획서 빨간 글씨 — 플랜에서 권장안 제시할 것)
1. **그리드 칸 크기** — 1칸 월드 단위 + 플레이어/오브젝트 스케일 기준 결정 필요.
2. **재료 스폰 방식** (4-J) — 1안/2안/혼합 중 택. UI·밸런스 영향 큼.
3. **공정 적용 방식** (4-C) — (a) 도구 가져와서 작업 / (b) 공정 선택 UI / (c) E 누르면 자동. 재미 vs 편의 트레이드오프. *기획자 선호 = (a) 도구 방식.*
4. **싱글 플레이 지원 범위** — 타임어택 싱글을 MVP에 포함할지.
5. **조기 종료 보너스** 채점 항목 사용 여부.
6. **던지기** 상호작용을 코어에 넣을지(레퍼런스상 협동 재미 핵심).

각 항목에 대해 클로드 코드는 선택지·트레이드오프·추천을 1~2줄로 제시할 것.
